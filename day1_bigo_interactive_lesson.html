DAY 1 • COMPLEXITY CLASSES

The complexity ladder

Each rung of this ladder shows how many operations your algorithm does as input n grows. The lower, the better.

O(1) Constant — doesn't grow at all
O(log n) Logarithmic — halves the problem each step
O(n) Linear — grows with input size
O(n log n) Linearithmic — fast sorts like merge sort
O(n²) Quadratic — nested loops, gets slow fast
O(2ⁿ) Exponential — doubles per input, unusable

Try this mental model:

If n = 1,000 — O(1) does 1 op, O(n) does 1,000, O(n²) does 1,000,000, and O(2ⁿ) is more operations than atoms in the universe.

DAY 1 • O(1) DEMO

Constant time — O(1)

No matter how large the array, accessing by index always takes exactly one step. The size of n doesn't matter at all.
JavaScript
const arr = [10, 20, 30, 40, 50]; // Access by index — always 1 step console.log(arr[0]); // O(1) ← instant console.log(arr[4]); // O(1) ← still instant // Doesn't matter if array has 5 or 5 million items

Click on any box to access it — notice it's always instant:

Click any element
O(1) Same speed regardless of array size. Excellent performance.
Other O(1) examples: push(), pop(), hash map lookup, variable assignment.
DAY 1 • O(n) DEMO

Linear time — O(n)

When you loop through every element once, the work grows directly with n. Double the input → double the work.
JavaScript
const arr = [10, 20, 30, 40, 50]; // Loop visits each element — n steps for (let i = 0; i < arr.length; i++) { console.log(arr[i]); // runs n times }
Press animate to see each step
O(n) Scales linearly. Acceptable for most tasks.
DAY 1 • O(n²) DEMO

Quadratic time — O(n²)

Nested loops multiply work. For each of the n outer iterations, you do n inner iterations. That's n × n = n² operations.
JavaScript
// Nested loops — n × n steps for (let i = 0; i < n; i++) { // n times for (let j = 0; j < n; j++) { // n times each console.log(i, j); // n² total ops } }

Watch how operations explode as n grows:

O(n²) Dangerous for large inputs. Look for nested loops in code reviews!
Common sources of O(n²): bubble sort, checking all pairs in an array, naive duplicate detection.
DAY 1 • GOLDEN RULES

The 4 golden rules

When calculating Big-O, follow these rules to simplify your answer:

1
Ignore constants
O(5n) = O(n)
Constant multipliers vanish at scale. Whether it loops 2× or 100× per element — if it's linear, it's O(n).
2
Add sequential ops
O(n)+O(n)=O(n)
Two separate loops one after another → still linear. You don't multiply — you add, then simplify.
3
Multiply nested ops
O(n)×O(n)=O(n²)
Loops inside loops multiply. This is why nested loops are expensive.
4
Keep dominant term
n²+n+1 → O(n²)
Drop smaller terms. At large n, n² dwarfs n and 1 completely. Only the fastest-growing term matters.

Space examples
// O(1) space — fixed variables let count = 0; // always 1 number in memory // O(n) space — grows with input const copy = [...arr]; // n elements in memory // O(n) space — hash map const map = new Map(); // stores up to n entries

JavaScript insight — arrays are objects!

In JS, arrays are specialized objects with numeric keys. This is why push() is O(1) — it adds a new key. But shift() is O(n) — it has to renumber every key.

DAY 1 • INTERVIEW FRAMEWORK

How to think in interviews

Top companies don't just want the right answer — they want to see your thought process. Always follow this 4-step framework:

1
Brute Force
Get it working first, any complexity
2
Find Bottleneck
Spot the slow part — nested loops? searches?
3
Pick Pattern
Hash map? Two pointers? Sliding window?
4
Optimize
Rewrite with better complexity
Two Sum — applying the framework
// STEP 1: Brute force — O(n²) function twoSum_slow(nums, target) { for (let i=0; i<nums.length; i++) // O(n) for (let j=i+1; j<nums.length; j++) // O(n) if (nums[i]+nums[j]===target) return [i,j]; } // STEP 3-4: Hash Map pattern → O(n) function twoSum_fast(nums, target) { const seen = new Map(); for (let i=0; i<nums.length; i++) { // O(n) if (seen.has(target-nums[i])) // O(1) lookup! return [seen.get(target-nums[i]), i]; seen.set(nums[i], i); } }
The hash map converts an O(n) inner search into an O(1) lookup — turning O(n²) into O(n). This is the #1 optimization pattern in interviews.
